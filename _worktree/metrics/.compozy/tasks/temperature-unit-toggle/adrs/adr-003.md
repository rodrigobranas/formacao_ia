# ADR-003: Pure Conversion Module in lib/units.ts

## Status

Accepted

## Date

2025-06-25

## Context

The PRD requires instant conversion of temperature, wind, precipitation, and pressure across all dashboard surfaces with **consistent rounding** (F4). Today, `frontend/src/lib/format.ts` holds display formatters that assume Celsius and metric units. Mixing conversion math into formatters would blur responsibilities and make unit tests harder to isolate.

The backend always serves Celsius/metric raw values; all conversion is client-side per ADR-001.

## Decision

Create **`frontend/src/lib/units.ts`** with pure conversion functions and unit-label helpers. **`format.ts` delegates to `units.ts`** by accepting a `UnitSystem` parameter (default `'metric'` for backward compatibility during migration).

Conversion rules:

| Measurement | Metric (stored) | Imperial (display) | Rounding |
|-------------|-----------------|--------------------|----------|
| Temperature | ┬░C | ┬░F | Whole degrees (`Math.round`) |
| Wind speed & gusts | km/h | mph | Whole (`Math.round`, factor 0.621371) |
| Precipitation | mm | in | 2 decimal places |
| Pressure | hPa | inHg | 1 decimal place (factor 0.02953) |

## Alternatives Considered

### Alternative 1: Extend format.ts only

- **Description**: Add conversion inline inside existing `formatTemperature`, `formatMeasure`, etc.
- **Pros**: Fewer files; all display logic in one place.
- **Cons**: Mixes math and presentation; harder to test conversion factors independently; format.ts grows unwieldy.
- **Why rejected**: Separation keeps rounding rules testable in isolation and matches the project's `lib/` utility pattern.

### Alternative 2: Combined units.ts with all formatting

- **Description**: Move all formatters into `units.ts`; deprecate `format.ts`.
- **Pros**: Single module for all unit-aware output.
- **Cons**: Large refactor of unrelated formatters (percent, cardinal, weekday); exceeds feature scope.
- **Why rejected**: YAGNIŌĆöonly unit-sensitive formatters need to change; percent/cardinal/time formatters stay in `format.ts`.

## Consequences

### Positive

- Conversion factors and rounding live in one tested module.
- Display components call formatters with `unitSystem` from contextŌĆöno inline math in JSX.
- Hourly curve and daily range bar can call the same `convertTemperature()` for positioning and labels.

### Negative

- Every unit-sensitive formatter signature gains a `unitSystem` parameter (or uses a wrapper).
- Two files to maintain for weather display formatting.

### Risks

- **Inconsistent rounding if a component bypasses formatters**: Mitigation: Impact Analysis lists every surface; code review checks for raw `data.current.temperature` in JSX.
- **Floating-point edge cases on pressure/precip**: Mitigation: unit tests with known conversion pairs (1013 hPa ŌåÆ 29.9 inHg).

## Implementation Notes

- Export `UnitSystem` type: `'metric' | 'imperial'`.
- Export `convertTemperature`, `convertWind`, `convertPrecipitation`, `convertPressure` returning `number | null` (null in ŌåÆ null out).
- Export label helpers: `temperatureUnitLabel`, `windUnitLabel`, `precipitationUnitLabel`, `pressureUnitLabel`.
- `format.ts` functions that are unit-sensitive accept optional `unitSystem` defaulting to `'metric'`.
