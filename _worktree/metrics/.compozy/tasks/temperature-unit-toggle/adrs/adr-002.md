# ADR-002: React Context for Unit Preference Propagation

## Status

Accepted

## Date

2025-06-25

## Context

The unit toggle affects six or more display surfaces (hero, quick stats, hourly, daily, detailed metrics) that are composed deep in the component tree under `WeatherDashboardPage`. Prop drilling would require threading `unitSystem` through every intermediate component. A standalone hook without shared state would risk desync between components if each instance read localStorage independently.

The PRD requires instant, global consistency—all in-scope readings must update in the same interaction frame when the toggle changes.

## Decision

Adopt **React Context** with a `UnitPreferenceProvider` wrapping the dashboard tree and a `useUnitPreference()` hook exposing:

- `unitSystem: 'metric' | 'imperial'`
- `setUnitSystem(system)` for programmatic updates
- Persistence via `localStorage` key `wx:units` (shape `{ temp: 'c' | 'f' }`, matching the design reference)

The provider lives at the page level (`WeatherDashboardPage`), so the toggle in the top bar and all weather cards share one source of truth.

## Alternatives Considered

### Alternative 1: Prop drilling from WeatherDashboardPage

- **Description**: Page reads preference hook once and passes `unitSystem` as a prop to each card.
- **Pros**: Explicit data flow; no Context API.
- **Cons**: Requires updating every intermediate wrapper; brittle as new cards are added.
- **Why rejected**: Too many touch points for a cross-cutting preference; Context is the established React pattern for global UI state that does not belong in the server payload.

### Alternative 2: Standalone hook per component

- **Description**: Each display component calls `useUnitPreference()` independently, each subscribing to the same localStorage key.
- **Pros**: No provider wrapper; components are self-contained.
- **Cons**: Requires custom sync mechanism (storage events or external store) to guarantee same-frame updates across all consumers; easy to introduce stale reads.
- **Why rejected**: Context provides atomic re-render of all consumers on toggle without extra sync infrastructure.

## Consequences

### Positive

- Single toggle updates all in-scope components in one React render cycle.
- New cards can opt in by calling `useUnitPreference()` without page-level prop changes.
- Provider boundary is clear: only dashboard content needs unit awareness.

### Negative

- Adds a Context provider to the page tree (minimal overhead).
- Components using the hook must render inside the provider (test setup must wrap accordingly).

### Risks

- **Missing provider in tests**: Components fail or throw if hook used outside provider. Mitigation: export a test helper wrapper or default context value for tests.
- **Over-broad re-renders**: All context consumers re-render on toggle. Mitigation: acceptable for ~6 components; payload is already in memory so no fetch cost.

## Implementation Notes

- Provider initializes from `localStorage` on mount; defaults to `'metric'` (Celsius) when key is absent or parse fails.
- Map `{ temp: 'c' }` → `'metric'`, `{ temp: 'f' }` → `'imperial'`; wind/precip/pressure follow the active system automatically (no separate wind toggle in MVP).
- `setUnitSystem` writes to localStorage synchronously before updating state.
