# ADR-002: react-i18next as Frontend i18n Library

## Status

Accepted

## Date

2025-06-25

## Context

The i18n feature requires instant locale switching, JSON translation resources for two locales, and integration with a React 19 + Vite SPA. The PRD mandates no full page reload on switch, browser persistence, and dynamic `lang` attribute updates. ADR-001 scoped the product approach but deferred library selection to TechSpec.

## Decision

Adopt **react-i18next** with **i18next** as the frontend i18n stack:

- Initialize i18next in a dedicated config module and wrap the app with `I18nextProvider` in `main.tsx`.
- Store translations as flat JSON files (`pt-BR.json`, `en.json`) bundled at build time (no lazy loading for MVP).
- Use `i18n.changeLanguage()` for instant switching; sync `document.documentElement.lang` on language change.
- Persist explicit user choice via `localStorage` (custom detector or `i18next-browser-languagedetector` with `localStorage` cache only — no browser-language auto-detection on first visit; PT-BR remains hard default per PRD).

## Alternatives Considered

### Alternative 1: react-intl (FormatJS)

- **Description**: ICU message format with React context provider.
- **Pros**: Strong Intl integration; mature formatting primitives.
- **Cons**: Heavier setup for a flat string inventory; less common in Vite SPA tutorials used by learners; separate formatting layer from translation keys.
- **Why rejected**: react-i18next aligns with ADR-001 research direction and offers simpler JSON-key workflow for ~100–150 static strings.

### Alternative 2: Custom hook + JSON (no library)

- **Description**: `useLocale()` hook reading from static JSON objects.
- **Pros**: Zero dependencies; full control.
- **Cons**: Reinvents interpolation, persistence wiring, context propagation, and test utilities; poor signal for learners studying production i18n patterns.
- **Why rejected**: Training sandbox should demonstrate industry-standard tooling, not a bespoke string map.

### Alternative 3: react-i18next (Selected)

- **Description**: i18next core with React bindings, JSON resources, `changeLanguage()` API.
- **Pros**: Widely adopted; JSON resources; instant switching; ecosystem plugins; good Vite compatibility.
- **Cons**: Adds two runtime dependencies (`i18next`, `react-i18next`); requires provider setup.
- **Why selected**: Best fit for two-locale SPA with persistence and learner-facing codebase clarity.

## Consequences

### Positive

- Standard pattern learners can reference in documentation and job contexts.
- Single `t()` function available in components and non-React modules (services, lib).
- Built-in support for interpolation and fallback locale (`pt-BR`).

### Negative

- Bundle size increases slightly (~15 KB gzipped combined).
- All components using strings must adopt `useTranslation()` or receive translated props.

### Risks

- **Hydration/flash of wrong language**: Mitigation — read `localStorage` synchronously before first render in i18n init.
- **Test complexity**: Mitigation — export test helper wrapping `I18nextProvider` with forced locale.

## Implementation Notes

- Install: `npm install i18next react-i18next` in `frontend/`.
- Config file: `frontend/src/i18n/config.ts`.
- Locale files: `frontend/src/i18n/locales/pt-BR.json`, `frontend/src/i18n/locales/en.json`.
- Default/fallback language: `pt-BR`.
- Supported languages: `['pt-BR', 'en']`.
