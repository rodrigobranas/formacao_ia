# ADR-003: API Label Localization via PT-String Remap

## Status

Accepted

## Date

2025-06-25

## Context

The backend embeds user-facing labels in weather API responses: WMO condition labels (~28 codes), UV categories, and air-quality category labels/descriptions — all authored in Brazilian Portuguese in `backend/src/services/weather-codes.ts`. The PRD explicitly excludes backend localization and `Accept-Language` negotiation. The frontend must display these dynamic labels in the active locale without changing the API contract.

Three strategies were evaluated: translate by WMO/AQI code on the frontend (ignore API label strings), remap API PT labels to translations, or add a `lang` query parameter to the backend.

## Decision

Localize **API-sourced labels on the frontend by remapping the PT label strings** returned in the JSON payload:

- Introduce `translateApiLabel(label: string): string` in `frontend/src/i18n/translate-api-label.ts` that looks up the PT string in an i18n key namespace (`apiLabels.*`) and returns the translated value for the active locale; if no mapping exists, return the original label unchanged.
- Apply this helper at display boundaries (hero condition, daily summaries, UV label, air-quality label/description) — not in the service layer that fetches data.
- Keep backend responses unchanged; backend PT strings serve as stable lookup keys.
- Maintain a complete `apiLabels` section in both locale JSON files covering every label emitted by `weather-codes.ts` plus the simplified EN air-quality set agreed in TechSpec review.

## Alternatives Considered

### Alternative 1: Translate by WMO code / AQI threshold on frontend

- **Description**: Ignore `label` fields from API; derive display text from `code` or numeric thresholds using frontend maps keyed by code.
- **Pros**: Decoupled from backend wording changes; single source of truth by numeric code.
- **Cons**: Duplicates backend threshold logic; requires syncing code maps when backend adds codes; air-quality descriptions need separate threshold replication.
- **Why rejected**: Stakeholder preference for simpler integration using existing API label strings as keys.

### Alternative 2: PT-string remap (Selected)

- **Description**: Use backend PT labels as lookup keys into i18n translation tables.
- **Pros**: No backend changes; no code/threshold duplication; display layer stays thin.
- **Cons**: Fragile if backend renames a PT label without updating frontend keys; keys are Portuguese strings (unusual i18n pattern).
- **Why selected**: Explicit TechSpec decision; minimal diff; backend label set is stable and owned by the same repo.

### Alternative 3: Backend `lang` query parameter

- **Description**: Extend `/api/weather` to accept `lang=pt-BR|en` and return localized labels.
- **Pros**: Single label map in backend; frontend displays API strings directly.
- **Cons**: Violates PRD non-goal; expands scope to backend services, tests, and error handling.
- **Why rejected**: Out of scope per PRD; frontend-only localization is a deliberate training constraint.

## Consequences

### Positive

- Zero API contract changes; backend tests unaffected.
- Clear display-layer boundary for dynamic label translation.
- Learners see a realistic pattern for localizing third-party or legacy API strings.

### Negative

- `apiLabels` keys must stay in sync with `backend/src/services/weather-codes.ts` PT strings.
- Portuguese strings as i18n keys are non-idiomatic and harder to grep than semantic keys like `weather.clearSky`.

### Risks

- **Backend label rename breaks EN display**: Mitigation — unit test asserting all backend PT labels have `apiLabels` entries; document coupling in TechSpec Impact Analysis.
- **Duplicate frontend `weather-code.ts` labels drift**: Mitigation — daily forecast card should use API labels via remap where payload includes them, or consolidate to code-based lookup only for simplified group labels.

## Implementation Notes

- Example lookup: `t(\`apiLabels.${label}\`, { defaultValue: label })` or explicit map object to avoid dot-in-key issues with special characters — prefer a flat `Record<string, string>` re-exported from locale JSON under `apiLabels` object.
- Air-quality descriptions: simplified bilingual set (not CONAMA/EPA regulatory naming).
- `frontend/src/lib/weather-code.ts` group labels (`labelForCode`) should migrate to i18n static keys or API remap for consistency.
