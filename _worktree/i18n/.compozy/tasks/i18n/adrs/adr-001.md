# ADR-001: Full UI i18n with PT-BR Default and Flag Switcher

## Status

Accepted

## Date

2025-06-25

## Context

The weather dashboard ("Painel de clima") currently ships all user-facing copy in Brazilian Portuguese. Learners and reviewers who prefer English cannot use the product comfortably. The team wants bilingual support (PT-BR and EN) without over-engineering a training sandbox SPA. Stakeholders confirmed: PT-BR remains the default on first visit, the language switcher uses flag icons in the top bar, preferences persist across sessions, and translations cover the full interface plus weather condition labels and localized date/time formatting.

## Decision

Deliver **Approach A — Full UI i18n with localized formatting**:

- **Default locale**: PT-BR on first visit when no saved preference exists.
- **Supported locales**: PT-BR and EN only.
- **Language switcher**: Compact flag control (🇧🇷 / 🇺🇸) in the top bar, alongside existing unit toggles.
- **Persistence**: Remember explicit user choice in the browser across sessions; explicit choice always overrides the PT-BR default.
- **Translation scope**: All interface copy (navigation, search, buttons, empty states, loading labels, toasts, API status), weather condition labels (WMO codes), and localized date/time display. Temperature unit toggle (°C/°F) remains independent of language.
- **Out of scope for this decision**: URL-based locale routing, additional languages, backend message localization, translation management platform, RTL support.

## Alternatives Considered

### Alternative 1: Essential UI Only (MVP Minimum)

- **Description**: Translate only critical visible strings (search, errors, empty states); defer weather labels and Intl formatting.
- **Pros**: Fastest delivery; smallest initial translation surface.
- **Cons**: English mode feels broken—hero shows Portuguese weather labels while chrome is English; dates stay in one format.
- **Why rejected**: Inconsistent bilingual experience undermines trust in a demo product meant to showcase production craft.

### Alternative 2: UI + URL Locale Prefix

- **Description**: Full translation scope plus `/pt` and `/en` URL paths for shareable, SEO-friendly links.
- **Pros**: Better for public marketing sites; canonical locale in address bar; aligns with 2026 i18n best practices for indexable content.
- **Cons**: Adds routing complexity, redirect logic, and link-sharing edge cases inappropriate for a local dev training sandbox with a single dashboard view.
- **Why rejected**: Product is a learner sandbox, not a public marketing site; user explicitly prioritized simplicity over SEO routing.

### Alternative 3: Full UI i18n with Localized Formatting (Selected)

- **Description**: Complete interface and weather label translation with localized date/time via standard locale-aware formatting; flag switcher with browser persistence.
- **Pros**: Coherent EN and PT-BR experiences; matches existing top-bar patterns; minimal conceptual overhead for learners studying full-stack craft.
- **Cons**: Larger initial translation inventory; layout must tolerate text expansion in EN.
- **Why selected**: Best balance of completeness and simplicity for a two-locale training product.

## Consequences

### Positive

- Learners can evaluate the dashboard in English or Portuguese without code changes.
- Explicit language choice is visible, discoverable, and persistent—reducing repeat friction.
- Weather labels and formatted dates reinforce that i18n is a product concern, not a string-replacement afterthought.
- Scope stays bounded to two locales, avoiding premature globalization infrastructure.

### Negative

- Initial translation and QA effort covers the entire UI surface plus ~20 weather condition labels.
- Flag-only affordances can be ambiguous (flags ≠ languages); must pair with accessible text labels.
- PRODUCT.md brand voice remains PT-BR-first; English copy needs a separate voice pass to stay direct and informative.

### Risks

- **Text overflow in glass panels**: English strings may be longer than Portuguese. Mitigation: verify layout in both locales during QA; follow existing flex-based patterns from design guidelines.
- **Stale hardcoded strings**: Missed strings break immersion. Mitigation: systematic inventory of all user-visible copy before sign-off.
- **Flag icon accessibility**: Screen readers may not interpret emoji flags reliably. Mitigation: require visible or aria-exposed language names alongside flags.

## Implementation Notes

- Recommended library direction from research: react-i18next for Vite/React SPAs (framework-agnostic, JSON resources, browser language detection plugin, `changeLanguage()` for instant switching). Final tooling belongs in TechSpec.
- Backend API error payloads may remain language-neutral; user-facing error text should be resolved in the presentation layer by error code.
- Weather condition labels currently live in a static PT-BR map and must gain EN equivalents.
- `lang` attribute on the document root should reflect active locale for accessibility and screen readers.
